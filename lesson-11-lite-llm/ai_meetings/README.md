# מערכת ישיבות AI לקבלת החלטות פיתוח

מערכת ישיבות מרובת-סוכנים ל-Claude Code: כל פרסונה רצה כ**סוכן AI עצמאי
ומבודד** — עם הקשר משלה והמודל שלה — כך שהעמדות מתנגשות באמת, במקום שמודל
אחד יביים ויכוח עם עצמו. התוצר של כל ישיבה הוא **מסמך החלטה**: שורה
תחתונה, טבלת אפשרויות, המלצה מנומקת, מחלוקות אמיתיות ומשימות.

---

## התחלה מהירה

```
/run-meeting pm,architect,critic האם לבנות את הפיצ'ר כ-API או כ-CLI
```

או פשוט:

```
/run-meeting
```

בלי ארגומנטים — המערכת תציג את כל הפרסונות הזמינות, תשאל מה ההחלטה על
הפרק, ותמליץ על הרכב מתאים.

---

## מה הופך את זה לאמיתי (ולא להצגה)

1. **בידוד** — בסבב הראשון כל פרסונה מקבלת את אותו חומר ולא רואה את
   האחרות. עמדה שנוצרה בנפרד היא עמדה, לא תפקיד בתסריט.
2. **עיגון חובה** — כל טענה חייבת ציטוט או סעיף מהחומר. עצות שנכונות לכל
   פרויקט ("חשוב לבדוק", "כדאי לשקול ביצועים") אסורות בפרוטוקול.
3. **מחויבות** — כל פרסונה חייבת להמליץ על אופציה עם אחוז ביטחון. אסור
   לשבת על הגדר.
4. **עימות ממוקד** — סבב שני רק בין עמדות שבאמת מתנגשות: כל צד מקבל את
   טיעוני הצד השני מילה במילה ותוקף בטיעונים חדשים. מותר לשנות עמדה —
   בתנאי שמנמקים מה שכנע.
5. **תוצר, לא תמליל** — הישיבה מסתיימת במסמך החלטה קריא; הפלט המלא של
   הסוכנים נשמר כנספח.

## הפקודות

| פקודה | מה היא עושה |
|---|---|
| `/run-meeting <פרסונות> <נושא>` | ישיבת החלטה: מסגור איתכם → סבב עמדות עצמאיות במקביל → סינתזה ומפת מחלוקות → סבב עימות → מסמך החלטה |
| `/review-spec <נתיב> [שאלת מיקוד]` | סקירת אפיון: סוקרים עצמאיים במקביל → פסק דין (מוכן / מוכן בתנאי / לא מוכן) + טבלת פערים לפי חומרה; פער שנמצא ע"י 2+ סוקרים מסומן "מאומת" |
| `/end-meeting` | סוגרת: מאשרת את מסמך ההחלטה, מתייקת משימות ב-`action-items.md` והחלטות ביומן `decisions.md` |
| `/new-agent <שם> <תיאור>` | יוצרת פרסונה חדשה ב-`.claude/agents/` — סוכן אמיתי עם מודל משלו, זמין מיידית |

מילת המפתח `all` מזמנת את כל הפרסונות (מומלץ רק לנושאים רוחביים).

| פרסונה | מודל | העדשה שלה | מתי להזמין |

| `tester` | sonnet | מקרי קצה, בדיקות קבלה | לפני שמכריזים על דרישה כ"סגורה" |
| `critic` | opus | פרקליט השטן — הנחות סמויות | **כמעט תמיד** |
| `ux` | sonnet | חוויית משתמש, זרימות, חיכוך | כל דבר שמשתמש נוגע בו |

כל פרסונה באמת רצה על המודל שבטבלה — עומק היכן שנדרש, מהירות היכן שמספיק.

## מהלך הישיבה

1. **מסגור (איתכם):** ההחלטה, קריטריוני ההצלחה והאילוצים — ואישור שלכם
   לפני שהסוכנים יוצאים לדרך (סבבי סוכנים עולים טוקנים אמיתיים).
2. **סבב 1 — עמדות עצמאיות:** כל הפרסונות רצות במקביל, מבודדות. כל אחת
   קוראת את החומר בעצמה ומחזירה עמדה + ממצאים מעוגנים + תובנה לא-צפויה ⚡.
3. **סינתזה ועצירה:** המנחה מציג טבלת עמדות ומפת מחלוקות אמיתיות —
   **וההגה עובר אליכם**: להרוג אופציה, להוסיף אילוץ, או להמשיך.
4. **סבב 2 — עימות:** רק פרסונות שעמדותיהן מתנגשות מקבלות את טיעוני הצד
   השני ותוקפות/מגינות. בלי קונפליקט — בלי סבב, חוסכים טוקנים.
5. **מסמך ההחלטה:** שורה תחתונה, טבלת אפשרויות מול קריטריונים, המלצה
   מנומקת, מחלוקות שנשארו, משימות. אתם מאשרים → `/end-meeting`.

## הדגמת שימוש — צעד אחר צעד למשתמש מתחיל

התרחיש: אתם צוות שמפתח את **TaskFlow**, אפליקציית ניהול משימות, ואתם
מתלבטים אם לפתח את פיצ'ר ההתראות בגרסה הקרובה. בפרויקט מחכה מסמך אפיון
אמיתי — [specs/notifications-spec.md](specs/notifications-spec.md) —
והישיבה תרוץ עליו. (האפיון הוא טיוטה מציאותית עם פערים מכוונים, כדי
שלסוקרים יהיה מה למצוא.)

### שלב 0 — פתיחה

פתחו את Claude Code בתיקיית הפרויקט (`ai_meetings`). חומר רקע (כמו האפיון
ב-`specs/`) צריך להיות בתיקייה כדי שהסוכנים יוכלו לקרוא אותו בעצמם.

### שלב 1 — נסחו את ההחלטה כשאלה

> "האם לפתח את פיצ'ר ההתראות בגרסה הקרובה, או לדחות?"

לא "בואו נדבר על התראות" — שאלה מכריעה. זה הצעד שהכי משפיע על התוצאה.

### שלב 2 — בחרו הרכב עם קונפליקט

להחלטה הזו: `pm`, `ux`, `devops`, `critic` — אינטרסים שמתנגשים מעצם
הגדרתם. לא בטוחים? הריצו `/run-meeting` ריק וקבלו המלצה.

### שלב 3 — פתחו את הישיבה

```
/run-meeting pm,ux,devops,critic בהתבסס על specs/notifications-spec.md — האם לפתח את פיצ'ר ההתראות בגרסה הקרובה או לדחות
```

### שלב 4 — אשרו את המסגור

המנחה יציג את ההחלטה, הקריטריונים והאילוצים וימתין לאישורכם. זה המקום
להזין עובדות שרק אתם יודעים: "אין לנו שרת פוש", "יש חודש", "הפריסה על
IIS". **מה שלא תגידו כאן — הסוכנים לא ידעו.**

### שלב 5 — סבב העמדות רץ לבד

כל הפרסונות רצות במקביל (זה לוקח דקה-שתיים — סבלנות, הן באמת קוראות את
האפיון). תקבלו טבלת עמדות: מי ממליץ על מה, בכמה ביטחון, ומה הממצא החזק
של כל אחת — כולל תובנות ⚡ שסומנו כלא-צפויות.

### שלב 6 — העצירה: הרגע שלכם

המנחה מציג את מפת המחלוקות האמיתיות ומחזיר לכם את ההגה:

```
תורידו את האופציה של דחייה מוחלטת
critic: רד לעומק על ההנחה שמשתמשים רוצים התראות
המשיכו לסבב העימות
```

### שלב 7 — סבב העימות

רק הפרסונות שעמדותיהן התנגשו מקבלות את טיעוני היריב ותוקפות בטיעונים
חדשים. פרסונה ששינתה עמדה תסביר מה שכנע אותה — וזה מידע זהב עבורכם.

### שלב 8 — קבלו מסמך החלטה ואשרו

המנחה מגיש את המסמך: שורה תחתונה, טבלה, המלצה, מחלוקות, משימות. אתם
מאשרים או מתקנים — **ההחלטה תמיד שלכם** — ואז:

```
/end-meeting
```

הסטטוס הופך ל"מאושר", המשימות מתויקות ב-`action-items.md` וההחלטות ביומן
`decisions.md`.

### שלב 9 — אחרי הישיבה

- מסמך ההחלטה ב-`meetings/` — קריא בפני עצמו, הנספח שומר את כל העמדות.
- `action-items.md` — רשימת המשימות; סמנו `[x]` כשמשימה הושלמה.
- בישיבה הבאה המערכת תקרא את `decisions.md` ולא תפתח החלטות סגורות מחדש.

### רוצים לתרגל גם סקירת אפיון?

```
/review-spec specs/notifications-spec.md
```

האפיון מכיל פערים מכוונים — בין השאר: אין מצבי ריק ושגיאה, אין זרימת
הרשאות iOS, "בזמן אמת" לא מוגדר, אין התייחסות לפרטיות, ומדדי הצלחה בלי
שיטת מדידה. בדקו כמה מהם נתפסים — פער שכמה סוקרים עצמאיים מצאו לבד יסומן
"מאומת ×N", וזו בדיוק ההוכחה שהבידוד עובד.

### טעויות נפוצות של מתחילים

| ❌ אל תעשו | ✅ עשו |
|---|---|
| נושא כללי ("בואו נדבר על ביצועים") | שאלה מכריעה ("האם לעבור ל-cache עכשיו?") |
| להזמין את כל הפרסונות לכל ישיבה | 3–5 עם אינטרסים מתנגשים |
| לאשר את המסגור בלי להוסיף עובדות | להזין אילוצים שרק אתם יודעים |
| לדלג על העצירה שאחרי הסינתזה | להרוג אופציות ולכוון את סבב העימות |
| לשכוח `/end-meeting` | בלי סגירה — אין החלטות, אין משימות, אין זיכרון |

## איך מפיקים תוצאות מיטביות

### 1. נסחו החלטה, לא נושא

- ❌ `/run-meeting pm,architect ארכיטקטורה של האפליקציה`
- ✅ `/run-meeting pm,architect,critic האם לפצל למיקרו-סרביסים עכשיו או אחרי ההשקה`

### 2. בנו הרכב עם קונפליקט מובנה

- `pm` נגד `architect` — מהירות מול איכות תכנון
- `biz` נגד `tester` — יציאה לשוק מול כיסוי מלא
- `critic` נגד כולם — תמיד

פחות מ-3 — אין דיון; יותר מ-5 — רעש ועלות.

### 3. הצביעו על קבצים ספציפיים

הסוכנים קוראים את החומר בעצמם — תנו להם נתיבים מדויקים בנושא הישיבה.
כלל העיגון עושה את השאר: טענה בלי ציטוט מהחומר נפסלת.

### 4. הזינו עובדות במסגור

אילוצים שרק אתם יודעים (תקציב, תשתית, דדליין) משנים עמדות של סוכנים
שלמים. משפט אחד במסגור חוסך סבב שלם של תיקונים.

### 5. חפשו את ה-⚡ ואת שינויי העמדה

שני הדברים הכי שווים במסמך: תובנות שסומנו כלא-צפויות, ופרסונה שכתבה "מה
ששכנע אותי לשנות עמדה". שם נמצא מה שלא הייתם מקבלים משיחה רגילה עם מודל.

### 6. סגרו תמיד עם `/end-meeting`

יומן `decisions.md` הוא הזיכרון המוסדי — ישיבות עתידיות קוראות אותו במקום
את כל המסמכים, לא פותחות מחדש החלטות סגורות, ועלות הפתיחה נשארת קבועה
גם אחרי חמישים ישיבות.

### 7. הרחיבו את הספרייה לפי הצורך

```
/new-agent data-scientist מומחה דאטה ומדדים, חושב בניסויים ו-A/B
```

הפרסונה נרשמת כסוכן אמיתי וזמינה מיידית. העיקרון: עדשה אחת חדה ודעתנית.
פרסונה שמסכימה עם כולם — מוחקים.

```
ai_meetings/

├── README.md                        # המדריך הזה

│   ├── agents/                      # הפרסונות — סוכני Claude Code אמיתיים
│   │   └── <name>.md                # עדשה + מודל (opus/sonnet)

│       ├── run-meeting.md           # מנוע הישיבה: 2 סבבים + סינתזה
│       ├── review-spec.md           # סקירת אפיון עם פסק דין ופערים
│       ├── end-meeting.md           # סגירה: אישור מסמך + תיוק
│       └── new-agent.md             # יצירת פרסונה חדשה
├── specs/
│   └── notifications-spec.md        # אפיון לדוגמה — ההדגמה רצה עליו

    ├── YYYY-MM-DD-<slug>.md         # מסמכי החלטה, בעברית

    └── action-items.md              # משימות מצטברות מכל הישיבות
```
